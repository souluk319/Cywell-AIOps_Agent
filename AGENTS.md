# CAS 작업 전 필수 지침서

이 문서는 Codex가 KOMSCO-AIOps_Agent 작업을 시작하기 전에 반드시 읽고 따라야 하는 프로젝트 작업 프로토콜이다. 목적은 코드를 많이 쓰는 것이 아니라, 사용자의 의도를 제품 경험으로 고정하고 검증 가능한 상태로 만드는 것이다.

## 0. 절대 원칙

- 배포는 사용자가 명시적으로 지시하기 전까지 하지 않는다.
- 기존 사용자가 만족한 UI/동작은 보존 대상이다. 단, 새 위치로 이동한 기능은 옛 위치에서 제거한다.
- “기능이 존재한다”는 완료가 아니다. 사용자가 자연스럽게 이해하고 쓸 수 있어야 완료다.
- verifier가 기존 계약을 강제하더라도 새 제품 방향과 충돌하면 verifier를 고친다.
- 최종 보고 전에는 테스트 로그, 스크린샷, 자가검수, 검수 에이전트 결과를 모두 대조한다.

## 1. 작업 시작 전 잠금

작업을 시작하기 전에 아래 4가지를 먼저 고정한다.

```text
목표:
완료 조건:
검증 방법:
하지 않을 것:
```

UI 작업이면 추가로 아래 표를 먼저 작성한다.

```text
기능 소유권:
- Header:
- Sidebar:
- Body:
- Composer:
- Settings/Popover:

유지할 것:
-

이동할 것:
-

제거할 중복:
-
```

이 표 없이 UI 코드를 먼저 치지 않는다.

## 2. CAS 챗봇 UI 기본 소유권

CAS v0.1.x 기준 기본 소유권은 다음과 같다.

```text
Header:
- 화면 전환: 채팅 / 상황 / 근거 / 다음 확인 / 시뮬레이션
- 분석 대상 설정
- 튜토리얼
- 언어 전환
- CAS 닫기

Sidebar / Wing:
- 새 대화
- 대화 기록
- 대화 검색
- 자동 저장 상태
- 설정
- 접기 / 펼치기

Body:
- 현재 선택된 화면의 본문
- 채팅 메시지
- RCA 상황 / 근거 / 다음 확인 / 시뮬레이션 결과

Composer:
- 질문 입력
- 추천 질문 +
- Ask / Troubleshooting 모드
- 전송 / 중지
```

이 소유권을 어기는 중복 버튼을 만들면 실패다. 예를 들어 새 대화가 Sidebar로 이동했으면 Header의 새 대화는 제거해야 한다.

## 3. UI 변경 절차

UI 변경은 아래 순서로만 진행한다.

1. 현재 기준 화면을 먼저 확인한다.
2. 기능 소유권 표를 만든다.
3. 유지할 것과 제거할 중복을 분리한다.
4. 구현한다.
5. verifier에 “있어야 할 것”과 “있으면 안 되는 것”을 모두 추가한다.
6. visual screenshot을 직접 확인한다.
7. 검수 에이전트에게 회귀 검토를 맡긴다.
8. 검수 결과를 반영한다.

다음 상태는 완료로 보고하지 않는다.

- 기능은 있지만 중복 버튼이 있다.
- 텍스트가 좁은 영역에서 부대낀다.
- 숨겨야 할 설명이 화면 공간을 차지한다.
- 버튼의 위치나 의미를 사용자가 추론해야 한다.
- visual verifier는 통과하지만 스크린샷이 어색하다.

## 4. 챗봇 기본 편의 체크리스트

CAS 챗봇 기능은 최소한 아래를 만족해야 한다.

- 입력 중 Enter 전송, Shift+Enter 줄바꿈
- 전송 버튼은 입력창 내부의 아이콘 버튼
- 실행 중 중지 버튼 표시
- 추천 질문은 고정 영역이 아니라 필요할 때 여는 popover
- Ask / Troubleshooting은 선택된 현재 모드만 표시
- 스트리밍 중 자동 스크롤
- 사용자가 위로 스크롤하면 자동 스크롤 해제
- 아래로 이동 버튼 제공
- 답변은 Markdown이 깨지지 않아야 함
- 내부 provider/debug 문구를 사용자 답변처럼 노출하지 않음
- 빈 새 대화는 저장하지 않음
- 일반 대화는 자동 저장
- 대화 기록은 검색 가능

## 5. 검증 기준

UI 작업의 완료 조건은 최소 아래를 포함한다.

```text
필수 검증:
- npm run verify:console-plugin
- npx tsc --noEmit -p apps/console-plugin/tsconfig.plugin.json
- npm run verify:console:integration
- npm run verify:console:visual
- git diff --check
```

검증에는 반드시 부정 조건을 포함한다.

```text
예:
- Header에 conversation/sidebar/history/new-chat 중복이 없어야 한다.
- Panel 내부 split/fullscreen/workspace wrapper가 없어야 한다.
- Sidebar는 panel 내부가 아니라 sibling이어야 한다.
- Desktop panel width는 기존 기준을 유지해야 한다.
- 대화 row는 compact height를 넘지 않아야 한다.
```

## 6. 병렬 검수 원칙

UI나 제품 흐름을 바꾸는 작업은 혼자 완료 판정하지 않는다.

- 구현자: 코드 수정
- 검수 에이전트: 제품 흐름, 중복, 상식적 사용감 검토
- verifier: 자동 검증 기준 보강

검수 에이전트가 잡은 문제가 있으면 “추가 의견”으로 넘기지 말고 완료 전에 반영한다.

## 7. 보고 방식

최종 보고에는 아래만 짧게 남긴다.

```text
현재 판단:
핵심 변경:
검증 결과:
남은 리스크:
branch / head sha:
```

“다 됐다”는 말은 visual screenshot을 직접 확인하고, 검수 에이전트 결과까지 반영한 뒤에만 쓴다.
